
# Project Discovery & Context Mapping

Este comando analisa automaticamente o projeto e gera um briefing técnico completo em formato modular.

É **OPCIONAL** e pode ser executado:
- Uma vez no início do projeto
- Quando ADRs mudarem significativamente
- Quando novo frontend Lovable for adicionado
- Quando arquitetura backend mudar

## Comportamento Incremental

Este comando é **não-destrutivo**:
- Se arquivos do briefing já existem: atualiza incrementalmente
- Se não existem: cria do zero
- Sempre preserva informações existentes

## Estrutura Gerada

```
docs/tech-context/
├── project-briefing.md              # Índice mestre + resumo
└── briefing/
    ├── critical-rules.md            # Regras não-negociáveis
    ├── adrs-summary.md              # ADRs consolidadas
    ├── backend-conventions.md       # Convenções de código
    ├── frontend-lovable.md          # Mapeamento Lovable (se existe)
    └── tech-stack.md                # Stack tecnológico
```

---

## Fase 1: Análise de ADRs (Condicional)

### 1.1 Detectar ADRs

```bash
# Procurar por pasta de ADRs
paths_to_check = [
  "docs/tech-context/adr/",
  "docs/adr/",
  "adr/"
]
```

- Se **NENHUMA** pasta existe: pular esta fase completamente
- Se existe: prosseguir com análise

### 1.2 Processar ADRs

Para cada arquivo `.md` encontrado na pasta de ADRs:

1. **Ler o arquivo completo**
2. **Extrair informações:**
   - Número da ADR (ex: ADR-001, ADR-015)
   - Título
   - Status (Aceito, Proposto, Rejeitado, Superseded)
   - Contexto/Background
   - Decisão principal
   - Convenções obrigatórias (se houver)
   - Consequências
   - Categoria (inferir: database, api, code-org, security, etc.)
   - Impacto (alto/médio/baixo - inferir pela criticidade)

3. **Identificar regras críticas:**
   - Procurar por palavras-chave: "obrigatório", "sempre", "nunca", "proibido"
   - Marcar ADRs com impacto "alto"

### 1.3 Consolidar ADRs

- Agrupar por categoria (database, api, code-organization, testing, security, etc.)
- Ordenar por impacto (alto → médio → baixo)
- Criar índice com links

---

## Fase 2: Mapeamento de Frontend Lovable (Condicional)

### 2.1 Detectar Frontend

Procurar por pastas comuns:
```bash
frontend_paths = [
  "apps/frontend/",
  "frontend/",
  "client/",
  "web/",
  "packages/frontend/",
  "src/frontend/"
]
```

Para cada pasta encontrada:
- Verificar se contém `package.json`
- Verificar se é React/TypeScript (dependencies: react, typescript)

### 2.2 Verificar se é Lovable

Padrões que indicam origem Lovable:
- Comentários com "lovable.dev" ou "Generated by Lovable"
- Estrutura típica: `src/components/`, `src/pages/`, `src/lib/`
- Presença de shadcn/ui (`components/ui/`)
- Vite config

Se **NÃO** for Lovable detectável:
- Documentar como "Frontend Manual/Custom"
- NÃO tentar mapear mocks (assumir que código está correto)

### 2.3 Mapear Mocks (se Lovable)

Invocar agente `@lovable-backend-mapper`:

```
Solicite ao @lovable-backend-mapper que:
- Escaneie todos os arquivos .tsx/.jsx em src/
- Identifique padrões de mock data:
  * useState com arrays/objetos hardcoded
  * Arquivos mock*.ts, fake*.ts, dummy*.ts
  * Dados inline em componentes
- Para cada mock encontrado, gere:
  * Localização (arquivo:linha)
  * Estrutura de dados (TypeScript interface)
  * Endpoint backend necessário (método + rota)
  * Checklist de integração
```

**Output:** Dados para preencher `frontend-lovable.md`

---

## Fase 3: Análise de Arquitetura Backend (Sempre executa)

### 3.1 Detectar Backend

Procurar por pastas:
```bash
backend_paths = [
  "apps/backend/",
  "backend/",
  "server/",
  "api/",
  "packages/backend/",
  "src/backend/"
]
```

### 3.2 Mapear Estrutura de Pastas

Para a pasta de backend encontrada, escanear:
- `src/controllers/` ou `src/routes/` → HTTP handlers
- `src/services/` → Business logic
- `src/repositories/` ou `src/dal/` → Data access
- `src/models/` ou `src/entities/` → Domain models
- `src/shared/` → Código compartilhado
  - `enums/` → ENUMs (IMPORTANTE!)
  - `types/` → TypeScript types
  - `schemas/` → Validation schemas
  - `utils/` → Helpers
- `src/tests/` ou `tests/` → Testes

### 3.3 Identificar Convenções

**Localização de ENUMs:**
- Procurar por arquivos `*.enum.ts` em toda a base
- Identificar onde a MAIORIA está (ex: `/shared/enums/`)
- Marcar como convenção se > 80% estão na mesma pasta

**Padrões de arquitetura:**
- Repository Pattern? (existem pastas `repositories/` E `services/`)
- Service Layer? (existe pasta `services/`)
- MVC? (existe `controllers/`, `models/`, `views/`)

**Nomenclatura:**
- Analisar 5-10 arquivos para identificar padrão:
  - kebab-case vs camelCase vs PascalCase
  - Sufixos (.controller.ts, .service.ts, etc.)

### 3.4 Detectar Stack Tecnológico

Ler `package.json` do backend:

**Framework Web:**
- express, fastify, nest, koa, hapi

**Database/ORM:**
- prisma, typeorm, drizzle, sequelize, mongoose

**Validação:**
- zod, joi, yup, class-validator

**Testes:**
- jest, vitest, mocha

**Runtime:**
- Verificar engines.node no package.json

## Fase 4: Geração dos Arquivos do Briefing

### 4.1 Criar pasta (se não existe)

```bash
mkdir -p docs/tech-context/briefing/
```

### 4.2 Gerar `project-briefing.md` (Índice Mestre)

Template completo com:
- 📊 Status do Projeto (resumo executivo)
- 📚 Índice de arquivos do briefing
- 🎯 Guia de uso por tipo de feature
- 🔄 Instruções de manutenção

**Tamanho alvo:** ~150 linhas

### 4.3 Gerar `briefing/critical-rules.md`

**Conteúdo:**
- Top 3-5 regras mais críticas das ADRs
- Convenções obrigatórias (ENUMs, Repository Pattern, etc.)
- Checklist de conformidade

**Importante:** Este arquivo será COPIADO integralmente para todo `context.md`

**Tamanho alvo:** ~80 linhas

### 4.4 Gerar `briefing/adrs-summary.md`

Se ADRs existem:
- Índice por categoria
- Cada ADR com:
  * Título, status, impacto
  * Decisão resumida
  * Convenções obrigatórias
  * Link para ADR completa

Se NÃO existem ADRs:
- Criar arquivo vazio com nota: "Nenhuma ADR definida ainda"

**Tamanho alvo:** ~400 linhas (escalável)

### 4.5 Gerar `briefing/backend-conventions.md`

**Conteúdo:**
- Estrutura de pastas (ASCII tree)
- Nomenclatura de arquivos/classes/métodos
- Padrões de código (controller/service/repository)
- Exemplos de código

**Tamanho alvo:** ~150 linhas

### 4.6 Gerar `briefing/frontend-lovable.md`

**Condicional:** SOMENTE se frontend Lovable detectado

**Conteúdo:**
- Status geral (X mocks pendentes)
- Para cada mock:
  * Componente (arquivo:linha)
  * Mock atual (código)
  * Endpoint necessário
  * Interface TypeScript
  * Checklist de integração (backend + frontend)

Se NÃO há frontend Lovable:
- NÃO criar este arquivo

**Tamanho alvo:** ~250 linhas (escalável com número de mocks)

### 4.7 Gerar `briefing/tech-stack.md`

**Conteúdo:**
- Runtime (Node.js versão)
- Framework web
- Database + ORM
- Bibliotecas principais
- Versões importantes

**Tamanho alvo:** ~100 linhas

### 4.8 Gerar `CLAUDE.md` (raiz do projeto)

**Sempre executado.** Criar ou sobrescrever `CLAUDE.md` na raiz do projeto.

Usar os dados coletados nas fases anteriores para preencher o template abaixo.
Substituir todos os placeholders `<...>` com valores reais do projeto.

```markdown
# <nome-do-projeto>

> Framework synapos v<versão> · Gerado por `/setup:discover` em <data ISO 8601> · Não editar manualmente

---

## Projeto

| Campo | Valor |
|---|---|
| **Nome** | <nome do projeto extraído de package.json ou inferido> |
| **Stack** | <framework web> + <database/ORM> + <runtime> |
| **Backend** | <caminho da pasta backend detectada> |
| **Frontend** | <caminho da pasta frontend ou "Não detectado"> |
| **Testes** | <framework de testes detectado ou "Não configurado"> |

---

## Regras Críticas do Projeto

> Extraídas de `docs/tech/architecture.md`

<listar as top 3-5 regras críticas identificadas no projeto — uma por linha, formato: `- **Regra:** descrição`>

Para a lista completa: `docs/tech/architecture.md`

---

## ADRs Ativas

> <adr_count> ADRs registradas em `docs/tech/adr/`

<listar ADRs com impacto alto, formato: `- **ADR-NNN** — Título (categoria)`>

<Se nenhuma ADR: "Nenhuma ADR definida. Consulte docs/tech/adr/">

---

## Estrutura do Framework

| Diretório | Conteúdo |
|---|---|
| `.synapos/core/` | Protocolos core do Framework Synapos |
| `.synapos/squads/` | Squads instanciados e seus agents |
| `.synapos/core/commands/` | Comandos de ativação dos modos |
| `docs/tech/` | Stack, ADRs e documentação técnica do projeto |
| `docs/business/` | Estratégia, personas e contexto de negócio |
| `docs/.squads/sessions/` | Estado das features em desenvolvimento |
| `docs/_memory/` | Configuração e memória do projeto (company, preferences) |

```

**Regras de preenchimento:**
- `<nome-do-projeto>`: extrair de `package.json` → campo `name`, ou do `project-briefing.md`, ou inferir do nome da pasta raiz
- Stack: máximo 60 caracteres (ex: `NestJS + Prisma + PostgreSQL`)
- Regras críticas: copiar apenas o texto da regra, sem blocos de código extensos
- ADRs: listar apenas as de impacto alto; se todas forem médias/baixas, listar as 3 primeiras
- Se algum campo não foi detectado: usar `"Não detectado"` (nunca inventar)

## Fase 5: Validação e Finalização

### 5.1 Verificar Integridade

- Todos os arquivos criados com sucesso?
- Links entre arquivos funcionam?
- Nenhum arquivo vazio (exceto casos válidos)?

### 5.2 Gerar Resumo

Contar:
- Total de ADRs analisadas
- Total de regras críticas identificadas
- Frontend Lovable? (Sim/Não)
- Total de mocks identificados (se aplicável)
- Total de linhas geradas

### 5.3 Informar ao Humano

```
✅ Project Discovery concluído com sucesso!

📁 Arquivos gerados:
- CLAUDE.md (raiz do projeto — bootstrap do framework)
- docs/tech-context/project-briefing.md (índice mestre)
- docs/tech-context/briefing/critical-rules.md
- docs/tech-context/briefing/adrs-summary.md
- docs/tech-context/briefing/backend-conventions.md

📊 Resumo da Análise:
- ADRs analisadas: X
- Regras críticas: X

- Mocks identificados: X [se aplicável]
```

⛔ **IMPORTANTE: Após informar o humano, PARE. NÃO prossiga automaticamente para /start ou qualquer outro comando. AGUARDE feedback do humano.**

## Ferramentas Utilizadas

Durante a execução, você DEVE usar:

- **Glob:** Para encontrar arquivos (ADRs, componentes, package.json, etc.)
  ```
  Glob pattern="docs/**/adr/*.md"
  Glob pattern="**/package.json"
  Glob pattern="apps/frontend/**/*.tsx"
  ```

- **Read:** Para ler conteúdo de arquivos
  ```
  Read file_path="docs/tech-context/adr/001-database.md"
  Read file_path="apps/backend/package.json"
  ```

- **@lovable-backend-mapper:** Para análise detalhada de frontend (se Lovable)
  ```
  Invocar via @ no contexto quando necessário
  ```

- **Write:** Para criar/atualizar arquivos do briefing
  ```
  Write file_path="docs/tech-context/project-briefing.md" content="..."
  Write file_path="docs/tech-context/briefing/critical-rules.md" content="..."
  ```

---

## Tratamento de Erros

### Se ADRs malformadas:
- Avisar ao humano: "ADR X está malformada, pulando"
- Continuar com outras ADRs
- Documentar no final quais foram puladas

### Se frontend sem padrão claro:
- Documentar como "Frontend Manual/Custom"
- NÃO tentar mapear mocks (evitar falsos positivos)

### Se estrutura backend não reconhecida:
- Perguntar ao humano:
  ```
  Detectei backend mas estrutura de pastas é não-convencional.

  Pasta raiz do backend: [caminho detectado]

  Por favor, me informe:
  1. Onde ficam os controllers/routes?
  2. Onde ficam os services?
  3. Onde ficam os ENUMs/types compartilhados?
  ```

### Se package.json não encontrado:
- Avisar: "Backend sem package.json detectado, análise limitada"
- Documentar o que foi possível inferir da estrutura de pastas

## Modo Verbose (Opcional)

Se o humano passar flag `--verbose`:
```
/discover --verbose
```

Mostrar output detalhado:
- Cada ADR sendo processada
- Cada arquivo sendo escaneado
- Cada mock sendo identificado
- Cada arquivo sendo gerado

Caso contrário (padrão):
- Mostrar apenas progresso resumido
- Reportar apenas erros/avisos

---

## Regras de Ouro

1. **Nunca assumir:** Se não encontrar algo, perguntar ao humano
2. **Ser incremental:** Não sobrescrever informações existentes sem razão
3. **Ser conservador:** Na dúvida sobre se é mock Lovable, NÃO marcar
4. **Ser explícito:** Links entre arquivos devem funcionar
5. **Ser resiliente:** Erros em uma fase não devem parar todo o processo
